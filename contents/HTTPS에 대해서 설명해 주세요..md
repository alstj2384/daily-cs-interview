# HTTPS에 대해서 설명해 주세요.
## 정제한 답변
HTTPS는 HTTP 메시지가 평문으로 오가기 때문에 통신 과정에서 발생할 수 있는 보안 문제를 해결하기 위해 고안되었습니다. 통신 과정에 TLS 보안 계층을 추가하여 안전한 통신이 수행되도록 돕습니다.
HTTPS 연결이 맺어지기 위해서는 TLS handshake가 수행되어야 합니다. TCP 3-way handshake로 신뢰성 있는 연결을 세우고, TLS handshake가 수행되는데요, TLS handshake가 끝나면 클라이언트와 서버는 같은 대칭키를 공유한 상태가 되고, 이후 HTTP 메시지는 이 대칭키로 암호화되어 통신합니다.
TLS Handshake는 서버 인증과 키 합의가 수행됩니다. 서버는 미리 자신의 공개키와 도메인 정보를 CA에 제출하고, CA는 신원을 확인한 뒤 그 정보를 CA의 개인키로 서명한 인증서를 발급합니다.
TLS handshake가 시작되면 서버는 클라이언트에 해당 인증서를 전달하고, 클라이언트는 브라우저에 내장된 CA 공개키로 서명을 검증합니다. 검증에 성공하면 "인증서 안의 공개키가 그 도메인의 것"을 보증할 수 있게 됩니다.
키 합의는 검증된 공개키를 이용합니다. 클라이언트는 통신에 쓸 대칭키를 서버의 공개키로 암호화해서 보내고, 서버는 자신의 개인키로 복호화해 대칭키를 얻습니다. 이 시점 이후부터 대칭키 기반의 암호화 통신이 시작됩니다.


## 공부한 내용 · 생각
### 대칭키, 개인키는 각각 어디서 저장되지?
개인키는 파일로 두거나 시크릿 매니저에 맡겨서 관리한다고 한다. 파일로 둔다면 권한을 엄격히 설정하여 보호해야 한다.
